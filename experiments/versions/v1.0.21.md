# v1.0.21 - Bounded range result guard

Date: 2026-06-21

## Goal

Continue the explicit optimization loop without losing correctness evidence. During range investigation, a small `0..f` window showed that the current native range path could save private keys outside the requested window.

## Root Cause

The GPU score kernel only checked `id <= rangeMax` in range mode. The actual candidate produced by the current iterator also includes the per-round `+G` offset, so later rounds could still accept and save keys after the requested finite range ended.

Current range mode still uses the original `+G` iterator path. This release fixes out-of-window result acceptance; it does not claim complete non-duplicate/non-skipping traversal yet.

## Code Change

- `kernels/kernel_profanity.cl`
  - Added a `rangeRoundOffset` argument to `profanity_score_matching`.
  - Range score checks now reject candidates when `id + rangeRoundOffset > rangeMax`.
- `Dispatcher.cpp`
  - Refreshes `rangeRoundOffset` before each score dispatch.
  - Tightens finite range exit logic to stop once the current reachable window is exhausted.
- `scripts/test-range-runtime.ps1`
  - New runtime regression script for bounded range output.
  - Verifies the known all-zero last-16 upward suffix-8 hit still reports private key `...2fcf6a3`.

## Rejected During This Pass

- Initial-point stride spacing (`id * deviceSize`) was tested conceptually and briefly prototyped, then removed.
- Reason: with the current `+G` iterator, large initial spacing makes low-window known hits unreachable within normal runtime. A real non-duplicate traversal needs a matching GPU iterator step, not only different initialization or CPU reconstruction.
