# v1.0.22 - Correct finite range coverage accounting

Date: 2026-06-22

## Goal

Continue the explicit optimization loop toward 400 MH/s while keeping the evidence honest. The user reported large duplicate counts in fixed-digit/range mode, and previous range MH/s numbers were still based on raw GPU work rather than unique newly covered private-key offsets.

## Root Cause

The current range kernel initializes work items at different offsets, but each completed dispatch advances every lane by only `+G`. That means later rounds slide across almost the same candidate window instead of advancing by the full device batch. v1.0.21 prevented saving out-of-window results, but speed sampling still counted raw GPU work and therefore overstated useful range coverage.

## Code Change

- `Dispatcher.cpp` / `Dispatcher.hpp`
  - Added range offset accounting helpers.
  - Added a per-device range sample watermark.
  - Range speed samples now count only newly covered offsets inside the user window.
  - Range exit now uses the coverage watermark instead of the old raw-round condition.
  - Saved result lines now omit `score=` and end with `time=...`.
- `scripts/test-range-runtime.ps1`
  - Verifies saved result format is `prefix/suffix/address/private/time`.
  - Verifies `score=` is not written.
  - Verifies the small finite range no longer reports duplicate raw GPU work as KH/MH after corrected accounting.

## Important Finding

With corrected accounting, the old range16 60-second figure around 370 MH/s was not a true unique-coverage rate. The v1.0.22 range16 verification reports about `7 H/s` of newly covered offsets after the first batch, proving that the next real optimization needs a GPU iterator stride redesign such as advancing each lane by `deviceSize * G` per round with matching CPU reconstruction.

Do not repeat earlier micro-tweaks as the main path:

- Keccak state initialization shortcut: no stable gain.
- Single-target suffix coarse filter: no stable gain.
- Fused iterate+score kernel: no stable gain.
- Constant-memory `negativeGx` / SHA table experiments: no meaningful gain.
- Initial-point stride spacing without changing GPU iteration: incorrect for known low-window hits.
