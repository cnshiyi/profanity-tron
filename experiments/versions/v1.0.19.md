# v1.0.19 - Correct range private-key reconstruction

Date: 2026-06-21

## Goal

Correct the v1.0.18 range stride risk before doing further performance work.

During post-release audit, v1.0.18 was found to have changed CPU-side range private-key reconstruction to use `round * deviceSize + id`. That would only be correct if the OpenCL iterator also advanced by `deviceSize * G` per round. The current kernel advances each initialized point by `+G` each round, so the CPU reconstruction must keep the same formula.

## Changes

- Restored `Dispatcher.cpp` range private-key reconstruction to:
  - `foundId << counterShift` for the initialized range offset.
  - `+G` per round via `addScalar64(seed, round + 1)`.
- Restored result queue and save-path signatures so they do not carry a misleading stride field.
- Kept the launcher behavior fixes from v1.0.18:
  - Random fixed-digit starts align the selected low digit window.
  - Blank direction is randomized once and reused by auto-continued windows.
  - Stop disables auto-continue.
- Extended `scripts/test-range-planner.ps1` so future changes fail if `round * stride` reconstruction returns without a matching kernel step change.

## Decision

Ship this correction before continuing optimization. Do not use v1.0.18's `round * deviceSize + id` range reconstruction as a baseline unless the OpenCL iterator is changed consistently.
