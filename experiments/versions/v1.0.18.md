# v1.0.18 - Range continuation and random digit alignment

Date: 2026-06-21

## Goal

Fix the fixed-digit/range behavior before continuing performance work:

- A completed 16-digit upward window must advance to the next window.
- Random digit count must affect the generated range, even when the initial private key is blank.
- Direction should be independently selectable and should also apply to randomly generated starts.
- Blank direction should randomize once at task start, not once per auto-continued window.
- Stopping the task must prevent auto-continue from launching another child process.

## Code Changes

- `Dispatcher.cpp` range candidate reconstruction now uses `round * deviceSize + id` as the linear range offset.
- Finite range mode now stops after the current window is covered instead of running `counterMax` full rounds and repeatedly scanning the same small window.
- Range result queue entries now carry the device stride used when the GPU generated the hit, so saved private keys are reconstructed from the same stride.
- The launcher now aligns random fixed-digit starts:
  - Upward direction fills the selected low digit window with `0`.
  - Downward direction fills the selected low digit window with `f`.
  - The random prefix outside that low digit window is preserved.
- Direction blank is randomized once at the beginning of the task and reused by auto-continued windows.
- Stop disables range auto-continue before killing the current process.

## Persistent Test

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\test-range-planner.ps1
```

The test records and verifies:

- `digits=16`, all-zero start, upward direction ends at `...ffffffffffffffff`.
- The next upward 16-digit window starts at `...00010000000000000000`.
- Random fixed-digit upward starts align low selected digits to `0`.
- Random fixed-digit downward starts align low selected digits to `f`.
- Direction is reused across auto-continued windows.
- Stop disables auto-continue.
- Explicit `<8`, `=8`, and `>8` digit branches remain present in the launcher.

## Blocked Local Runtime Check

Direct execution of the freshly built local binary was blocked by Windows Application Control:

```text
Program 'shiyi.exe' failed to run: An Application Control policy has blocked this file
```

This release therefore does not claim a 1-minute performance result or the 400 MH/s target. The next performance pass should use a policy-allowed release binary or a signed build.

## Decision

Ship this as a meaningful behavior fix, not a documentation-only release. The 400 MH/s goal remains open.
